# EnvMason（境匠）产品需求文档

> 文档状态：I00 已接受基线 v0.1
> 适用阶段：产品定义与增量开发基线
> 目标平台：macOS、Windows、Linux
> 产品形态：本地 CLI 核心 + AI/Skill/MCP 控制入口
> 许可证：MIT

## 1. 文档目的

本文档定义跨平台开发环境管理平台的产品目标、范围、用户场景、功能需求、安全边界、非功能需求和验收原则。它是后续设计、开发、测试和版本验收的统一依据。

开发工作必须遵循以下原则：

1. 核心能力只有一套实现，CLI、Skill、MCP 和未来 GUI 都调用同一个核心引擎。
2. 默认只读，所有写操作必须先扫描、再生成可审查的 Plan、经用户确认后执行并验证。
3. 优先编排操作系统和生态已有的包管理器，不重新实现软件下载、依赖解析和软件仓库。
4. 每个增量必须可以独立运行、测试、验收和回退；未通过当前增量验收，不进入下一增量。
5. “最新”不等于“最适合”，系统必须同时考虑稳定版、LTS、项目约束、EOL 和用户保留策略。
6. 自动识别清理候选不等于自动删除；卸载、换源、提权、系统服务修改属于高风险行为，必须采用更严格的策略与确认机制。

## 2. 产品概述

### 2.1 产品定位

EnvMason（中文名：境匠，CLI 命令：`envmason`）是一个开发者工作站生命周期管理平台，用统一方式管理一台设备上的开发工具、语言运行时、包管理器、容器环境、SDK 和开发 Agent。配置、缓存、状态和数据目录遵循各平台原生规范，具体路径和迁移规则在相关实现增量中确定。

平台解决以下核心问题：

- 用户不知道本机安装了哪些开发环境，以及实际生效的是哪一个版本。
- 同一个工具可能由多个管理器重复安装，造成 PATH、架构或默认版本冲突。
- 单纯升级到最新版可能破坏已有项目，而手工判断保留和清理成本很高。
- 新设备配装需要重复执行大量平台相关命令，结果难以复现。
- 不同平台的包管理器和版本管理器缺少统一报告、统一策略与统一操作入口。
- AI 可以辅助管理环境，但需要确定性的执行核心、安全约束和审计记录。

### 2.2 产品愿景

让开发者能够用一条命令或一次自然语言交互完成以下闭环：

> 发现环境 → 理解状态 → 生成安全计划 → 执行变更 → 验证结果 → 必要时恢复

### 2.3 核心价值

- **统一可见性**：跨平台、跨管理器展示完整环境清单。
- **可解释建议**：说明为什么升级、保留、迁移或建议删除。
- **安全执行**：所有变更先形成确定计划，高风险操作明确确认。
- **可复现配装**：开发画像表达意图，锁文件记录精确结果。
- **AI 友好**：稳定的结构化接口供 Skill、MCP 和其他 Agent 调用。

## 3. 产品目标与非目标

### 3.1 产品目标

| 编号 | 目标 | 衡量方式 |
|---|---|---|
| G-01 | 生成可信的本机开发环境报告 | 已支持工具的发现准确率达到既定测试基线，报告包含来源与路径 |
| G-02 | 判断版本状态与支持状态 | 区分当前、稳定版、LTS、项目要求与 EOL |
| G-03 | 生成安全、可解释的更新计划 | 每个动作包含原因、风险、前置条件和预期结果 |
| G-04 | 执行选择性或批量更新 | 支持 dry-run、确认、日志、验证和中断恢复 |
| G-05 | 提供新设备配装能力 | 相同 Profile 可在目标平台生成等价环境，并产生 Lock |
| G-06 | 发现可安全清理的旧环境 | 自动计算严格满足安全规则的候选；实际卸载始终生成 R3 Plan 并由用户明确确认 |
| G-07 | 管理软件源与镜像 | 能检测、评估、切换并恢复受支持的源配置 |
| G-08 | 支持 AI 管理入口 | AI 只能通过受控能力生成和应用计划，不能绕过策略 |

### 3.2 非目标

首个稳定版候选路线明确不做以下事项：

- 不开发新的通用包管理器或软件仓库。
- 不保证任意第三方安装器都能完全回滚。
- 不自动卸载来源不明、系统关键或仍被项目引用的软件。
- 不让 AI 直接执行任意 Shell、PowerShell 或管理员命令。
- 不在首版覆盖所有 Linux 发行版、所有 CPU 架构和所有开发工具。
- 不在首版提供图形界面、云端账户同步或团队设备管理后台。
- 不负责备份用户项目代码、数据库数据、容器卷或私有密钥。
- 不绕过软件许可证、企业安全策略、系统签名与包校验机制。

### 3.3 版本范围解释

- I00–I41 构成首个稳定版候选的总体路线，I41 表示首个稳定版候选。
- 不要求等待 I41 才公开发布；I08 可形成 macOS 只读预览版，后续里程碑按 `0.x` 逐步发布。
- I42 及后续增量属于 v1.x 完整能力扩展。
- 产品愿景和目标工具类别不代表当前版本已经支持；实际能力只以已验收增量、发布说明和兼容矩阵为准。

## 4. 用户与典型场景

### 4.1 目标用户

| 用户 | 主要需求 |
|---|---|
| 个人开发者 | 了解和更新本机环境，减少版本冲突 |
| 多语言开发者 | 管理多个 Node、Java、Python、Go 等运行时 |
| 新入职工程师 | 根据团队标准快速配装新设备 |
| 运维/平台工程师 | 安装容器、Kubernetes、IaC 和云平台工具 |
| 测试工程师 | 快速建立自动化测试与浏览器测试环境 |
| 团队技术负责人 | 定义可复现、可审计的团队开发画像 |
| AI 编程工具用户 | 让本地 Agent 在明确权限和策略下维护环境 |

### 4.2 核心用户故事

- 作为开发者，我希望看到所有 Node 安装实例、来源和当前生效版本，以便发现 NVM 与 Homebrew 冲突。
- 作为 Java 开发者，我希望知道 Maven、Gradle、终端和 IDE 分别使用哪个 JDK，以免构建行为不一致。
- 作为维护者，我希望把仍被项目引用的旧版本标记为必须保留，避免批量更新后项目无法运行。
- 作为新设备用户，我希望选择“前端 + Java 后端”，预览安装内容后一次完成配装。
- 作为网络环境受限的用户，我希望检测当前软件源是否可达、是否及时同步，并在确认后切换到可靠镜像。
- 作为 AI 用户，我希望用自然语言生成操作计划，但在卸载、换源和提权之前仍由我确认。

## 5. 产品形态与运行模式

### 5.1 本地 CLI 模式

本地 CLI 是产品的基础入口，必须在没有 AI、Node、Python 或 Java 运行时的干净设备上可启动。CLI 提供扫描、报告、规划、执行、验证、恢复和画像管理能力。

基础命令域应保持稳定，具体参数在技术设计阶段确定：

- 环境扫描与报告
- 版本和生命周期比较
- 更新、安装、清理和换源计划
- 计划预览与应用
- 结果验证与操作历史
- Profile 解析与 Lock 生成
- 诊断和适配器状态

### 5.2 AI/Skill/MCP 模式

AI 模式是 CLI 核心之上的受控入口，负责：

- 将自然语言需求转换为明确的扫描或规划参数。
- 解释报告、冲突、版本策略和风险。
- 协助选择 Profile、版本策略和替代工具。
- 调用已生成的计划并展示确认项。

AI 模式必须遵守以下稳定安全约束：

| 编号 | 约束 |
|---|---|
| AI-001 | AI 不得替用户确认或绕过确认执行写操作 |
| AI-002 | AI 不得降低核心或适配器给出的风险等级 |
| AI-003 | AI 不得修改计划内容后沿用原 Plan ID、签名或确认状态 |
| AI-004 | AI 不得调用核心未声明的任意 Shell、PowerShell、提权或系统命令 |
| AI-005 | AI 不得隐藏安装来源、提权要求、重启要求、许可证提示或失败状态 |

## 6. 功能需求

### 6.1 系统与平台发现

| 编号 | 需求 |
|---|---|
| FR-001 | 识别操作系统、版本、CPU 架构、主机架构与兼容层 |
| FR-002 | 识别当前用户、Shell、PATH 顺序和关键环境变量，但不得泄露敏感值 |
| FR-003 | 识别系统可用的包管理器、版本管理器和权限能力 |
| FR-004 | 区分原生架构与兼容架构安装，例如 Apple Silicon 原生与 Rosetta |
| FR-005 | 扫描失败时保留部分结果，并标明失败原因与置信度 |

### 6.2 开发工具清单

| 编号 | 需求 |
|---|---|
| FR-010 | 发现工具的全部已知安装实例，而不只读取 PATH 中第一个实例 |
| FR-011 | 记录工具统一 ID、显示名称、版本、路径、架构、安装来源和管理器 |
| FR-012 | 标识当前生效版本、默认版本和非激活版本 |
| FR-013 | 发现同一工具由多个管理器安装的情况 |
| FR-014 | 识别 PATH 遮蔽、损坏软链接、命令与包数据库不一致等冲突 |
| FR-015 | 支持按类别、工具、管理器、状态和风险过滤报告 |

目标工具类别包括：

- 基础工具：Git、SSH、CMake、Make、编译器、证书工具。
- 语言运行时：Node.js、Java、Python、Go、Rust、Ruby、.NET。
- 生态工具：npm、pnpm、Yarn、Maven、Gradle、pip、Poetry、Cargo。
- 容器工具：Docker Desktop、Docker Engine、Docker Compose、Podman、Colima。
- 平台 SDK：Xcode/Command Line Tools、Windows SDK、Visual Studio Build Tools。
- 运维工具：kubectl、Helm、Terraform/OpenTofu、云平台 CLI。
- 开发 Agent：仅在后续扩展中纳入安装与更新。

这些类别描述产品目标，不构成当前版本支持承诺。v1 各阶段的实际范围以增量计划为准。macOS v1 第一条垂直切片聚焦系统与 PATH、Homebrew、Node/NVM/npm/pnpm、Java/jenv/Maven/Gradle，以及报告、建议、Plan 和安全执行基础；Docker、完整 DevOps 配装和更多语言工具按后续增量推进。

### 6.3 版本、支持状态和项目约束

| 编号 | 需求 |
|---|---|
| FR-020 | 获取工具的最新稳定版、最新 LTS、发布日期和 EOL 状态 |
| FR-021 | 版本信息必须记录数据来源、获取时间、缓存时间和可信度 |
| FR-022 | 网络不可用时使用未过期缓存；缓存过期时明确提示结果可能陈旧 |
| FR-023 | 扫描常见项目版本声明和构建文件，建立项目到运行时的引用关系 |
| FR-024 | 支持用户 Pin、保留、忽略更新和版本通道策略 |
| FR-025 | 输出“已是推荐版本、可更新、必须保留、已 EOL、存在冲突、未知”状态 |
| FR-026 | 不把语义上不可比较的版本强行排序，无法判断时必须降级为未知 |

### 6.4 环境报告

| 编号 | 需求 |
|---|---|
| FR-030 | 输出终端表格、人类可读 Markdown 和机器可读 JSON |
| FR-031 | 报告包含摘要、系统信息、工具清单、版本状态、冲突、风险和建议 |
| FR-032 | 每个建议包含原因、证据、置信度和可能影响 |
| FR-033 | 报告默认隐藏访问令牌、密码、私钥、完整代理认证信息等敏感数据 |
| FR-034 | 支持对两次扫描结果进行差异比较 |
| FR-035 | 报告 Schema 必须版本化，并保持已承诺字段的兼容策略 |

### 6.5 更新计划与执行

| 编号 | 需求 |
|---|---|
| FR-040 | 用户可为全部、类别或指定工具生成更新计划 |
| FR-041 | 更新计划包含动作、执行管理器、前置条件、风险、提权、下载量、重启要求和验证方式 |
| FR-042 | 支持 dry-run；dry-run 不得修改文件、包数据库、服务或源配置 |
| FR-043 | 应用计划前验证计划未过期、环境前置状态未发生关键变化 |
| FR-044 | 每一步执行后立即验证，并记录退出状态、标准化错误和必要日志 |
| FR-045 | 执行失败后停止依赖步骤，允许安全步骤继续或由策略决定中止 |
| FR-046 | 支持中断后从已验证的检查点继续 |
| FR-047 | 用户可以手动选择或排除具体更新项 |
| FR-048 | 更新完成后重新扫描，并给出计划前后差异和未解决问题 |

### 6.6 保留与清理

| 编号 | 需求 |
|---|---|
| FR-050 | 为每个可清理对象计算项目引用、激活状态、默认状态、依赖关系和用户 Pin |
| FR-051 | 只有管理器已知、无引用、非激活、非默认、非依赖且存在可用替代版本时，才可列为自动清理候选 |
| FR-052 | 来源不明、系统关键、判断不确定或恢复困难的对象只能建议，不得自动删除 |
| FR-053 | 清理计划必须单独列出预计释放空间与潜在影响 |
| FR-054 | 卸载、删除缓存和删除配置必须作为不同动作分别授权 |
| FR-055 | 删除前保存可恢复信息；无法恢复时必须明确标记 |

| 编号 | 需求 |
|---|---|
| FR-070 | 识别受支持管理器当前配置的软件源、镜像和代理状态 |
| FR-071 | 检测可达性、延迟、证书、同步新鲜度和完整性保障能力 |

| 编号 | 需求 |
|---|---|
| FR-090 | 每个计划具有唯一标识、创建时间、环境摘要和不可歧义的动作集合 |
| FR-091 | 保存执行前快照、逐步执行结果、验证结果和执行后差异 |
| FR-092 | 对可逆动作提供恢复计划；对不可逆动作明确标注“仅尽力恢复”或“不可恢复” |

| FR-094 | 执行恢复前再次生成并确认恢复计划 |

## 7. 风险等级与确认策略

| 等级 | 示例 | 默认策略 |
|---|---|---|
| R0 只读 | 扫描、版本查询、报告 | 可直接执行 |
| R1 低风险可逆 | 更新本地缓存、生成 Lock | v1 仍需计划级或批量确认，不允许配置静默授权 |
| R2 常规变更 | 安装、升级普通开发工具 | 计划级确认 |
| R3 高风险 | 卸载、换源、修改默认运行时、系统服务 | 明确确认，必要时逐项确认 |
| R4 敏感系统变更 | 提权、驱动、注册表关键项、系统安全设置 | 单独确认；不适合首版开放的能力直接禁止 |

任何写操作都必须来自可审查且未过期的 Plan。任何动作的风险等级只允许被适配器或平台策略上调，不能由 AI、配置或用户画像静默下调；AI 不能代表用户作出确认。

系统可以自动计算清理候选，但实际卸载永远属于 R3，必须重新生成 Plan 并由用户明确确认；不支持 AI 或配置进行无人值守删除。

## 8. 统一数据模型概念

### 8.1 Tool

表示一个规范化工具，例如 `runtime.node`、`runtime.java`、`build.maven`。

### 8.2 Installation

表示 Tool 的一个具体安装实例，至少包含：

- 规范化工具 ID 与显示名称
- 版本原文与规范化版本
- 可执行文件或安装根目录
- 平台和 CPU 架构
- 安装来源和管理器
- 当前激活、默认、直接安装或依赖安装状态
- 项目引用和用户保留状态
- 数据来源、发现时间和置信度

### 8.3 VersionStatus

表示当前版本与稳定版、LTS、EOL 和项目约束的比较结果。

### 8.4 Finding

表示冲突、风险、异常或建议，包含证据、严重程度、置信度和推荐动作。

### 8.5 Plan 与 Action

Plan 是不可歧义的动作集合；Action 是最小执行单元，包含前置条件、依赖、风险、权限、执行器、验证器和恢复信息。

### 8.6 Profile 与 Lock

- Profile：声明用户希望获得的能力和版本策略。
- Lock：记录某次解析后各平台的精确实现和版本，用于复现与审计。

## 9. 平台支持策略

### 9.1 首个稳定版候选支持范围

| 平台 | 首个稳定版候选范围 | 后续或观察范围 |
|---|---|---|
| macOS | Apple Silicon；系统与 PATH；Homebrew；Node/NVM/npm/pnpm；Java/jenv/Maven/Gradle；报告、建议、Plan 和安全执行基础 | Intel、Mac App Store、Docker、完整 DevOps、更多语言与容器实现 |
| Windows | Windows 11、WinGet、PowerShell、基础 WSL 发现 | Scoop、Chocolatey、VS Installer、完整 WSL 管理 |
| Linux | Ubuntu LTS、apt | Debian 仅观察和测试，在建立独立验收矩阵前不声明正式支持；其后为 Fedora/dnf、Arch/pacman、SUSE/zypper、Snap/Flatpak |

首个稳定版候选不追求平台功能数量完全一致，而要求相同概念、报告和安全语义一致。每个具体版本只承诺已经通过对应增量验收的能力。

### 9.2 兼容性声明

每个发布版本必须包含：

- 支持的操作系统及最低版本。
- 支持的 CPU 架构。
- 支持的管理器版本范围。
- 已知限制和未验证组合。
- 报告 Schema 和 Profile Schema 版本。

## 10. 非功能需求

### 10.1 安全

- 默认扫描不得修改系统状态。
- 不记录或上传密码、令牌、私钥和凭据内容。
- 下载应使用官方或明确受信来源，并保留签名与校验机制。
- 提权范围必须最小化，禁止长期保存管理员密码。
- Profile、Lock、插件和 AI 请求都视为不可信输入并严格校验。
- 执行命令必须使用结构化参数，避免不安全的字符串拼接。

### 10.2 可靠性

- 扫描和计划生成必须幂等。
- 单个工具失败不能阻止其他工具报告。
- 执行步骤必须有超时、退出码、标准化错误和验证逻辑。
- 应对断网、下载失败、磁盘不足、权限不足、用户中断和需要重启的情况。

### 10.3 性能

- 快速本地扫描目标：常规开发机在 10 秒内产生基础清单。
- 包含远程版本查询的完整扫描目标：正常网络下 60 秒内完成，慢源可超时降级。
- 无依赖的探测任务应并发执行，但不得造成包管理器锁冲突。

以上为目标值，需在性能基线增量中通过真实测试确定最终阈值。

### 10.4 可测试性

- 核心模型、策略和计划器不得依赖真实系统命令才能单元测试。
- 适配器必须支持 fixture/模拟输出测试。
- 破坏性集成测试使用可重置虚拟机或专用测试设备。
- 容器不能替代涉及 GUI、UAC、Xcode、Docker Desktop 和系统服务的真实平台测试。

### 10.5 可观测性与隐私

- 默认仅保存本地日志。
- 如未来引入遥测，必须默认关闭或明确征得同意，并公开字段说明。
- 日志支持机器可读格式、关联计划 ID 和敏感信息脱敏。

### 10.6 可维护性

- 平台差异限制在适配器内部。
- 外部命令输出解析必须有版本化 fixture 测试。
- 版本和 EOL 数据源允许替换，并能记录来源健康状态。
- 核心公开 Schema 采用明确的兼容与迁移策略。

## 11. 测试与发布门槛

### 11.1 测试层级

1. 单元测试：模型、版本比较、策略、计划依赖和脱敏。
2. 契约测试：适配器输入输出与统一 Schema。
3. Fixture 测试：不同版本和语言环境下的管理器输出。
4. 集成测试：真实包管理器的只读与安全安装场景。
5. 端到端测试：扫描、计划、执行、验证和恢复闭环。
6. 虚拟机测试：Windows UAC、Linux 发行版和破坏性操作。
7. 手工验收：GUI 安装器、重启、许可证和系统服务场景。

### 11.2 增量发布门槛

每个增量进入下一阶段前必须满足：

- 需求和非目标明确。
- 实现方案、风险和验收标准明确。
- AI 已完成自检，维护者可按需检查 diff。
- 与变更风险相称的自动化及必要手工测试通过。
- 原有回归测试通过。
- 支持平台矩阵内的验收通过。
- 适用的文档和示例已同步；不适用项直接记录 N/A，不制造空交付物。
- 无未处置的高风险安全问题。
- 如果产生系统变更，已验证失败路径与恢复路径。
- 维护者已确认完成，或已预先授权在全部客观验收通过后自动进入下一增量。

## 12. 成功指标

首个公开版本建议使用以下指标：

- 已支持工具发现准确率和误报率。
- 版本状态判断准确率。
- 扫描成功率与平均耗时。
- 更新计划应用成功率。
- 更新后验证通过率。
- 自动清理候选的零已知误删目标。
- Profile 配装一次成功率和二次幂等成功率。
- 跨平台同一 Profile 的能力一致率。
- 用户取消高风险操作后系统保持原状态的成功率。

## 13. 主要风险与应对

| 风险 | 影响 | 应对策略 |
|---|---|---|
| 外部管理器输出变化 | 适配器解析失败 | 优先结构化输出、fixture 契约测试、能力探测 |
| “最新版”数据错误或陈旧 | 错误升级建议 | 多源校验、缓存时效、来源与置信度展示 |
| 多管理器重复安装 | PATH 和默认版本冲突 | 记录所有实例，先诊断再提供迁移计划 |
| 自动清理误判 | 项目不可用或数据丢失 | 极保守规则、项目引用扫描、明确确认、恢复信息 |
| 提权与命令注入 | 系统安全风险 | 参数化执行、最小权限、输入校验、禁止任意命令 |
| Linux 发行版碎片化 | 维护成本失控 | 分阶段支持，按适配器声明能力 |
| GUI 安装器与重启 | 自动化中断 | 检查点、继续执行、人工步骤状态 |
| 第三方镜像供应链风险 | 安装恶意或陈旧软件 | 可信清单、签名校验、默认官方源、可恢复 |
| AI 误解用户意图 | 执行错误动作 | AI 只生成受控计划，高风险动作由核心阻断和确认 |

## 15. 术语表

| 术语 | 定义 |
|---|---|
| Adapter | 对某个平台、管理器或工具的检测和操作适配层 |
| Inventory | 本机开发环境的规范化清单 |
| Finding | 根据清单和策略产生的问题、风险或建议 |
| Plan | 执行前生成且可审计的动作集合 |
| Action | 计划中的最小执行和验证单元 |
| Profile | 描述期望开发能力和版本策略的声明文件 |
| Lock | 描述精确包、版本、来源和平台解析结果的锁文件 |
| Pin | 明确要求保留或固定某个版本 |
| EOL | 官方停止支持或停止维护的生命周期状态 |
| Dry-run | 只生成或模拟计划，不改变系统状态 |
| Checkpoint | 可用于中断恢复或生成恢复计划的执行记录点 |
