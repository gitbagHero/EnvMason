# EnvMason AI 协作约定

本文件适用于仓库根目录及其全部子目录。

## 项目角色

- 项目维护者是产品决策者、范围与安全边界决策者和最终验收人。
- AI 是开发协作者，负责分析、设计建议、实现、测试、文档和代码自检。
- 不建立虚拟的产品、架构、开发、测试或项目管理角色。
- AI 可以提出建议，但不能代替维护者决定产品范围、安全边界、许可证或公开接口。

## 增量门禁

- 严格按照 `INCREMENTAL_DEVELOPMENT_PLAN.md` 的顺序，一次只实现一个最小增量。
- 开始实现前明确用户价值、范围、非范围、依赖、方案、风险、测试和验收标准。
- 当前增量的测试和验收未完成前，不得实现下一增量。
- 后续需求进入 Backlog；除非它是当前增量验收的必要条件，不得顺手扩大范围。
- 不适用的交付物直接标记为 N/A，不创建空代码、空 fixture 或空文档。
- 增量完成后由维护者确认，或按维护者预先授权在全部客观验收通过后进入下一增量。

## 安全边界

- 默认只读。
- 所有写操作必须来自确定性核心生成、可审查且未过期的 Plan。
- R0 可以直接执行；v1 中 R1 和 R2 需要计划级确认；R3 需要明确确认，必要时逐项确认；R4 需要单独确认，不适合首版开放的能力直接禁止。
- 自动清理只表示自动识别候选；实际卸载永远是 R3，并需要用户明确确认。
- AI 不能替用户确认、静默降低风险、修改 Plan 后复用原 Plan ID，或绕过核心执行任意 Shell、PowerShell、提权或系统命令。
- 安装、卸载、换源和系统修改的测试优先在可恢复环境中运行。
- 不得省略失败路径、执行后验证、恢复说明或敏感信息保护。

## 实现与验证

- 核心能力只实现一次，CLI、Skill、MCP 和未来 GUI 调用同一确定性核心。
- 不重新实现通用包管理器；通过有能力声明和统一输出的适配器编排现有管理器。
- 修改应最小、单一、可理解、可测试、可回退。
- AI 在交付前运行与风险相称的自动化测试并自检 diff，逐项核对验收标准。
- 测试失败时继续停留在当前增量，不能通过文档声明绕过失败。

## Git 规则

- 代码审查采用 AI 自检、自动化测试和维护者按需检查 diff。
- 项目早期允许在 `main` 上进行经确认的小步修改；不强制每次创建分支或 Pull Request。
- 高风险执行、安全边界、公开 Schema 和正式发布优先使用独立分支及 PR 式审查。
- 未经维护者明确授权，不提交、推送、创建 PR、打标签或发布版本。
- 不覆盖、回退或清理维护者尚未提交的修改。
