# EnvMason（境匠）敏捷增量开发计划

> 文档状态：I00 已接受基线 v0.1
> 配套需求：[PRODUCT_REQUIREMENTS.md](./PRODUCT_REQUIREMENTS.md)
> 交付策略：一次只实现一个最小增量；当前增量全部测试和验收通过后，才允许开始下一增量。

## 1. 拆解目标

本文档把产品需求拆解为可独立开发、测试和验收的最小增量。每个增量都必须提供一个可观察的新能力，避免同时铺开多个平台、管理器和高风险执行功能。

增量顺序遵循以下风险递进：

> 数据模型 → 只读发现 → 本地报告 → 远程版本比较 → 项目约束 → 计划 → 单工具执行 → 恢复 → 配装 → 清理 → 换源 → 跨平台 → AI

## 2. 敏捷执行规则

### 2.1 单增量工作流

每个增量依次完成：

1. 明确目标和用户价值。
2. 明确本次范围、非范围和依赖。
3. 明确最小实现方案、技术风险、测试方案和验收标准；长期影响架构、安全、兼容性或产品范围的决定记录到 `docs/PROJECT_DECISIONS.md`。
4. 完成实现，不夹带下一增量功能。
5. 运行与风险相称的自动化测试和必要的手工测试。
6. AI 自检变更并逐项对照验收标准，维护者按需检查 diff。
7. 由维护者确认完成，或按维护者的预先授权在全部客观验收通过后进入下一增量。

不要求 Sprint 会议、每日站会、评审会、回顾会、Story Point、燃尽图、工时统计、多人审批或角色签字。是否发布版本由里程碑和维护者决定，不要求每个增量独立发布。

### 2.2 在制品限制

- 同一时间只能有一个产品增量处于实现状态。
- 可以并行准备测试 fixture 或调研后续风险，但不能提前合并后续功能。
- 发现新的必要工作时，优先加入待办；只有当前增量无法验收时才纳入当前范围。
- 缺陷优先在引入它的增量中修复，不以“后续再处理”绕过质量门槛。

### 2.3 Definition of Ready

一个增量开始前必须满足：

- 用户价值和目标可用一句话说明。
- 范围内、范围外和依赖已明确。
- 验收标准可以被测试。
- 所需测试环境可获得。
- 不存在未决的高风险产品选择。

### 2.4 Definition of Done

一个增量完成必须满足：

- 所有验收标准通过。
- 实现具有相称的单元、契约、fixture、集成或端到端测试；不适用项直接记录 N/A。
- 全量回归测试通过。
- 不引入未处理的高等级安全问题。
- 受影响的 CLI 帮助、Schema、示例和用户文档已同步。
- 新能力已在该增量声明的支持环境中验证。
- 错误路径可理解，不输出秘密或令人误解的成功状态。
- AI 已完成自检，维护者已确认完成或已明确预先授权自动过门。

### 2.5 项目角色与 Git 流程

- 项目维护者是产品决策者和最终验收人；AI 是开发协作者，不是独立审批人。
- 所有待决策事项默认由维护者负责，不维护虚拟负责人字段。
- 代码审查采用 AI 自检、测试和维护者按需检查 diff。
- 项目早期允许在 `main` 上进行经确认的小步修改，不强制每次创建分支或 Pull Request。
- 高风险执行、安全边界、公开 Schema 和正式发布优先使用独立分支及 PR 式审查。
- 未经维护者明确授权，不提交、推送、创建 PR 或发布版本。

## 4. 增量详细拆解

### M0：工程与契约基线

#### I00：产品契约冻结

**用户价值**：维护者与 AI 对首版范围、术语、安全边界和协作方式形成统一理解。

**内容**：

- 将 EnvMason／境匠写入正式产品基线。
- 修正 v1、v1.x、目标工具类别和支持平台的范围表述。
- 修正批量更新覆盖关系，并为其建立独立最小增量。
- 统一自动清理和 R0–R4 确认语义。
- 将两份核心文档整理到仓库根目录。
- 创建最小 README、AGENTS、LICENSE、`.gitignore` 和 `docs/PROJECT_DECISIONS.md`。
- 记录 I00 的客观检查和维护者验收结果。

**不包含**：任何产品代码和真实环境探测。

**测试/验收**：

- 产品名称、版本范围、正式支持平台和 macOS 首条垂直切片表述一致。
- 批量更新不再错误映射到只读增量，并具有独立最小增量。
- R0–R4、AI 确认和自动清理权限边界无矛盾。
- 长期决策和未决事项的最迟决策增量已记录。
- 文档链接、需求编号、增量编号和依赖关系检查通过。
- 仓库不包含产品代码或 I01 功能。

**退出条件**：PRD v0.1 和增量计划 v0.1 被接受为开发基线。

#### I01：可运行的空 CLI

**用户价值**：项目第一次产生可安装、可识别版本的跨平台二进制。

**内容**：

- 建立 Go 模块、目录结构、构建和测试命令。
- CLI 支持帮助、版本、构建信息和标准退出码。
- CI 在 macOS、Windows、Linux 上完成编译和单元测试。

**不包含**：扫描、配置文件、网络请求和系统修改。

**测试/验收**：

- 三个平台构建成功。
- `help` 和 `version` 不依赖外部运行时。
- 未知命令返回非零退出码和可理解错误。
- 构建产物包含版本、提交和目标平台信息。

**依赖**：I00。

#### I02：统一清单 Schema 与 fixture 框架

**用户价值**：后续所有平台用同一种方式表达环境，避免数据模型分裂。

**内容**：

- 定义 System、Tool、Installation、Finding、SourceMetadata 的最小模型。
- 定义并版本化 JSON 输出 Schema。
- 建立 fixture、快照测试和 Schema 兼容测试框架。
- 提供一份手工构造的示例报告。

**不包含**：真实命令探测、版本比较、更新建议。

**测试/验收**：

- 合法示例通过 Schema 校验。
- 缺失必填字段、非法枚举和未知 Schema 版本被拒绝。
- 同一对象序列化结果稳定，快照测试通过。
- 模型允许表达同一工具的多个安装实例。

**依赖**：I01。

### M1：macOS 本地环境盘点

#### I03：macOS 系统只读探测

**用户价值**：用户可以获得本机基础系统信息，且扫描不会修改设备。

**内容**：

- 识别 macOS 版本、CPU/进程架构、Shell 和 PATH 条目。
- 识别 Apple Silicon、Intel 和 Rosetta 运行状态。
- 对环境变量进行白名单采集与脱敏。

**不包含**：Homebrew 和语言运行时探测。

**测试/验收**：

- 在 Apple Silicon 真实设备输出正确系统与进程架构。
- PATH 保持顺序且重复项可识别。
- 扫描前后文件系统和包管理器状态无变化。
- 环境中放置测试令牌时，报告和日志不出现其值。

**依赖**：I02。

#### I04：通用可执行文件发现器

**用户价值**：系统能说明命令实际从哪个路径生效，并发现 PATH 冲突。

**内容**：

- 按 PATH 顺序发现指定命令的全部实例。
- 记录真实路径、软链接和架构。
- 标记首个生效实例、重复路径和损坏链接。

**不包含**：把命令映射到包管理器、查询最新版。

**测试/验收**：

- fixture 中两个同名命令都被发现，首个实例标记正确。
- 软链接循环和损坏链接不会导致扫描崩溃。
- 包含空格和 Unicode 的路径正确处理。
- 权限不足的路径产生 Finding，而不是中止扫描。

**依赖**：I03。

#### I05：Homebrew 只读适配器

**用户价值**：用户能看到 Homebrew 安装位置、源、已安装包和过期状态。

**内容**：

- 发现 Homebrew 前缀、版本、架构和仓库远端。
- 使用结构化输出获取 formula、cask 和 outdated 数据。
- 将 Homebrew 包映射到统一 Installation。
- 记录命令失败、锁占用和结构化输出版本。

**不包含**：`brew update`、升级、卸载、换源和自动修复。

**测试/验收**：

- Homebrew 缺失时给出“未安装”，扫描仍成功。
- Intel 与 Apple Silicon 前缀 fixture 均解析正确。
- formula、cask 和依赖安装状态可区分。
- 测试确认未调用任何 Homebrew 变更命令。

**依赖**：I04。

#### I06：Node.js 生态只读适配器

**用户价值**：用户能看到 NVM 管理的全部 Node 版本及 npm/pnpm/Yarn 的真实来源。

**内容**：

- 发现系统、Homebrew 和 NVM 下的 Node 实例。
- 识别 NVM 默认别名和当前生效版本。
- 发现 npm、Corepack、pnpm 和 Yarn 的版本及路径。
- 报告多来源安装和 PATH 遮蔽。

**不包含**：安装 Node、修改 NVM alias、升级全局包。

**测试/验收**：

- 至少覆盖仅系统 Node、仅 NVM、多来源、无 Node 四类 fixture。
- 能表达 Node 22 被保留且另有默认版本的场景。
- npm/pnpm 与所属 Node 路径关系可追踪。
- NVM 未加载到当前 Shell 时仍能通过已知目录进行受控发现，失败则明确降级。

**依赖**：I05。

#### I07：Java 生态只读适配器

**用户价值**：用户能识别 JDK、jenv、JAVA_HOME、Maven 和 Gradle 使用版本不一致的问题。

**内容**：

- 发现系统 JDK、Homebrew JDK 和 jenv 已登记版本。
- 记录 `java`、`JAVA_HOME`、jenv global/local 的实际指向。
- 获取 Maven、Gradle 运行时所使用的 Java 版本。
- 输出版本选择冲突 Finding。

**不包含**：升级或删除任何 JDK，修改 jenv 配置。

**测试/验收**：

- 覆盖单 JDK、多 JDK、失效 jenv 链接和 Maven Java 不一致 fixture。
- 报告能区分 JDK 安装版本与 Maven 实际运行版本。
- 项目目录不存在时不产生虚假的 local 版本结论。
- 任何子命令失败只影响对应字段。

**依赖**：I06。

#### I08：macOS 首份综合 Markdown/JSON 报告

**用户价值**：用户第一次得到可读、可保存、可机器处理的完整报告。

**内容**：

- 汇总系统、PATH、Homebrew、Node 和 Java 数据。
- 输出终端摘要、Markdown 和 JSON。
- 支持按工具类别和严重程度过滤。
- 报告标注扫描时间、扫描范围、失败项和数据来源。

**不包含**：远程最新版、EOL、更新动作。

**测试/验收**：

- 三种输出对同一事实保持一致。
- JSON 通过 I02 Schema 校验。
- Markdown 在常见渲染器中结构清晰。
- 部分适配器失败时报告仍生成，并显著标注不完整。
- 本机手工核验首批工具清单无已知漏报或误报。

**依赖**：I07。

### M2：版本、项目约束与智能评估

#### I09：通用版本规范化与比较

**用户价值**：系统能可靠比较常见版本格式，不因格式差异给出错误结论。

**内容**：

- 支持 SemVer、Java 特性版本和常见厂商后缀的最小比较能力。
- 保留原始版本字符串和规范化结果。
- 对不可比较版本返回 Unknown，而不是猜测。

**不包含**：联网查询最新版。

**测试/验收**：

- 建立边界值、预发布、build metadata、厂商后缀测试集。
- 比较器满足反对称性和传递性测试。
- 无法解析的版本不会触发升级或清理建议。

**依赖**：I08。

#### I10：远程版本与 EOL 数据提供器

**用户价值**：报告能说明稳定版、LTS 和支持状态，并在离线时诚实降级。

**内容**：

- 为 Node 和 Java 接入首批可信版本数据源。
- 增加来源、获取时间、TTL、缓存和超时。
- 区分 Latest Stable、Latest LTS、EOL 和 Unknown。
- 提供离线缓存行为。

**不包含**：其他工具最新版、执行更新。

**测试/验收**：

- 正常网络、超时、无网络、损坏缓存和过期缓存测试通过。
- 报告显示来源与数据时间。
- 过期缓存不会显示为“已确认最新”。
- 外部数据异常不导致本地报告失败。

**依赖**：I09。

#### I11：项目版本引用扫描

**用户价值**：系统知道哪些运行时仍被本地项目使用，避免错误升级或删除。

**内容**：

- 在用户明确指定的目录扫描 `.nvmrc`、`.node-version`、`package.json engines`、`.java-version`、`.tool-versions`、`pom.xml` 和 Gradle 配置。
- 建立项目、工具、版本约束之间的引用关系。
- 默认不扫描用户整个磁盘，不读取依赖目录和秘密文件。

**不包含**：自动修改项目配置、解析所有构建系统的动态逻辑。

**测试/验收**：

- 每种首批文件格式具有正例、缺失字段和损坏文件测试。
- 忽略 `node_modules`、构建产物、版本库内部目录和用户排除路径。
- 同一项目冲突声明形成 Finding。
- 未指定项目根目录时不擅自全盘扫描。

**依赖**：I10。

#### I12：首版建议与冲突规则

**用户价值**：报告不只展示数据，还能解释应升级、保留或调查什么。

**内容**：

- 为 Node 和 Java 实现规则：已推荐、可升级、EOL、项目要求保留、多来源冲突、默认版本冲突。
- 每条 Finding 输出证据、原因、严重程度、置信度和建议。
- 用户 Pin 和忽略策略先以只读配置形式加入。

**不包含**：生成可执行计划、自动删除。

**测试/验收**：

- Node 22 被项目引用时，即使存在更新版本也不建议删除。
- 最新 Current 与 LTS 不混为一个推荐值。
- Maven 使用的 Java 与 Shell 不一致时给出可解释 Finding。
- Unknown 数据不会产生确定性更新结论。
- 规则决策表和单元测试覆盖全部分支。

**依赖**：I11。

### M3：安全更新闭环

#### I13：只读 Plan/Action 模型

**用户价值**：用户可以预览未来会做什么，而系统仍不会修改设备。

**内容**：

- 定义 Plan、Action、依赖、风险、前置条件、验证和恢复元数据。
- 将 Node 的一个更新建议转换为不可执行的预览计划。
- 计划具有唯一 ID、环境摘要和创建时间。

**不包含**：执行器、提权、真实安装。

**测试/验收**：

- 相同输入和策略生成语义一致的计划。
- 循环依赖、缺失验证器和未知风险等级被拒绝。
- 计划 JSON 通过版本化 Schema 校验。
- 预览命令经系统调用审计确认无写操作。

**依赖**：I12。

#### I14：通用受控执行器与操作日志

**用户价值**：产品具备执行一个低风险测试动作并留下完整审计记录的基础能力。

**内容**：

- 参数化进程执行、超时、取消、退出码和标准化错误。
- 本地计划日志和逐步状态机。
- 使用项目自带的无害测试动作验证执行框架。
- 日志脱敏和输出长度限制。

**不包含**：真实包升级和管理员权限。

**测试/验收**：

- 成功、失败、超时、取消和异常退出测试通过。
- 命令参数包含空格、Unicode 和特殊字符时不发生注入。
- 模拟令牌不会出现在日志。
- 进程中断后计划状态不是虚假的 Completed。

**依赖**：I13。

#### I15：单个 NVM Node 版本安装

**用户价值**：用户可通过计划安装一个明确指定的新 Node 版本，同时保留已有版本。

**内容**：

- 仅支持 NVM 已存在时安装明确版本。
- 生成、确认并应用单动作计划。
- 安装后验证版本存在，不自动切换默认版本。
- 记录安装前后差异。

**不包含**：安装 NVM、升级 npm/pnpm、修改 alias、卸载旧 Node。

**测试/验收**：

- dry-run 不产生新目录或 NVM 状态变化。
- 用户拒绝确认时保持原状态。
- 安装成功后新版本可被 NVM 列出，原 Node 22 保留。
- 下载失败、磁盘不足和用户中断产生正确失败状态。
- 二次执行不重复安装且验证通过。

**依赖**：I14。

#### I16：Node 默认版本切换

**用户价值**：安装新版本后，用户可以通过独立计划选择是否把它设为默认版本。

- 完成安装文档、快速开始、故障排查、安全说明和兼容矩阵。
- 冻结 v1 Schema，制定后续兼容与弃用策略。
- 发布候选版本并处理阻断问题。

**测试/验收**：

- I00–I40 的回归门槛全部通过。
- macOS、Windows 11、Ubuntu LTS 的支持矩阵测试通过。
- 无阻断级缺陷或未解释的数据迁移风险。
- 新用户能按文档完成扫描、计划、Base 配装和结果验证。

**依赖**：I40。

### M9：v1.x 完整能力扩展

M9 不属于首个稳定版的发布阻断范围，用于把 v1 已验证的模型逐步扩展到 PRD 的完整产品愿景。每项仍需严格顺序验收，不允许以“功能类似”为由批量合并。

#### I42：macOS 通用开发工具扩展

**用户价值**：macOS 报告覆盖 Git、CMake、Python、Go、Rust、Ruby、.NET、Docker 和常用生态工具。

**内容**：

- 为通用开发工具增加规范化 ID、只读适配器和版本来源。
- 区分系统自带、Homebrew、语言专用管理器和独立安装来源。
- 增加 Docker Desktop、Engine、Compose 的客户端/服务端状态区分。

**测试/验收**：

- 每种工具至少覆盖未安装、单来源、多来源和命令损坏 fixture。
- Docker Desktop 已安装但 Engine 未运行时状态准确。
- 系统自带工具不会被误判为可由 Homebrew 直接删除。

**依赖**：I41。

#### I43：macOS 受控选择性与批量更新

**用户价值**：用户可以从可靠发现的普通开发工具中选择、排除或批量更新，并得到统一验证结果。

**内容**：

- 首批仅处理核心已能可靠发现、规划和验证的 macOS 普通开发工具。
- 将多个独立更新动作组合为可选择和排除的 Plan，保持依赖顺序、风险分级、计划过期检查和逐项验证。
- 支持计划级批量确认；任何 R3/R4 动作仍按自身等级单独确认或禁止。
- 执行后重新扫描并输出每项真实状态和整体差异。

**不包含**：卸载、换源、系统组件、驱动、无人值守更新和跨管理器事务保证。

**测试/验收**：

- 用户可以选择、排除单项或确认一组符合条件的 R1/R2 更新。
- 任一动作失败时，依赖动作不会错误执行，其他动作状态不被伪装为成功。
- 环境关键状态变化后旧计划失效，不能静默应用。
- R3/R4 动作不能借由批量确认降低确认级别。
- dry-run、拒绝确认和无可更新项时不改变系统状态。

**依赖**：I42。

#### I44：MacPorts 共存影响与迁移建议

**用户价值**：用户能判断 MacPorts 是否仍承担不可替代的工作，并安全规划向 Homebrew 迁移。

**内容**：

- 只读发现 MacPorts 安装包、requested/dependency 状态、端口前缀和 PATH 影响。
- 将 MacPorts 包映射为规范化工具，与 Homebrew 和系统实例对比。
- 输出“可被 Homebrew 替代、暂不可替代、重复安装、未知”的迁移建议。

**不包含**：自动卸载 MacPorts 或删除 `/opt/local`。

**测试/验收**：

- 依赖包不会被当成用户主动安装包。
- 存在 MacPorts 独有能力或未知映射时拒绝给出自动删除结论。
- PATH、launch daemon 和配置引用都进入影响报告。
- 无 MacPorts 的设备扫描正常完成。

**依赖**：I43。

#### I45：Java 升级、默认版本与安全清理闭环

**用户价值**：用户可安装推荐 JDK、让 Maven/Gradle 使用目标版本，并清理无引用的旧 JDK。

**内容**：

- 支持选择 JDK 发行版、LTS/精确版本和安装管理器。
- 将安装、jenv 登记、默认切换、Maven/Gradle 验证拆成独立动作。
- 对旧 JDK 使用与 Node 相同的保守清理策略。

**测试/验收**：

- JDK 17/21 被项目引用时必须保留。
- JDK 安装成功但 Maven 仍使用旧版本时计划不算完成。
- 切换失败可以恢复原默认 JDK。
- 来源不明的 JDK 只产生迁移建议，不自动删除。

**依赖**：I44。

#### I46：Java Backend Profile 实际配装

**用户价值**：三平台用户可通过同一 Java Backend Profile 配装 JDK、Maven 和可选 Gradle。

**内容**：

- 为 macOS、Windows 和 Ubuntu 增加 Java 模块平台解析与 Lock。
- 执行安装、默认版本选择和构建工具验证。
- 提供最小示例项目进行端到端构建验收。

**测试/验收**：

- 三平台能够编译并运行相同最小 Java 示例。
- 二次配装幂等。
- 不同 JDK 发行版选择正确反映在 Lock 中。
- 许可证或来源要求在计划中明确展示。

**依赖**：I45。

#### I47：DevOps Profile 实际配装

**用户价值**：用户可选择一个容器实现，并配装 kubectl、Helm 和 IaC 工具。

**内容**：

- 分平台实现 Docker Desktop、Colima、Podman 或 Docker Engine 的受支持组合。
- 安装并验证 kubectl、Helm、Terraform/OpenTofu。
- 区分客户端安装成功、守护进程运行和基本容器验证。

**测试/验收**：

- 互斥容器实现不会被同时设为默认。
- 容器 hello-world、客户端/服务端版本和 Compose 能力验证通过。
- GUI、许可证、虚拟化、重启和提权要求在执行前展示。
- 无法使用硬件虚拟化时在预检阶段停止。

**依赖**：I46。

#### I48：Python、Go 与 QA Profile

**用户价值**：后端和测试用户可以组合安装 Python、Go 与浏览器自动化测试能力。

**内容**：

- 增加 Backend Python、Backend Go 和 QA 模块。
- 支持明确的运行时管理策略、虚拟环境工具和最小验证项目。
- QA 首批覆盖 API 测试基础和一种浏览器自动化实现。

**测试/验收**：

- 三个模块可以单独使用，也可组合且共享能力去重。
- 系统 Python 与用户管理 Python 不被混淆或错误覆盖。
- 最小 Python、Go 和浏览器测试示例通过。
- 浏览器二进制下载量和平台依赖在计划中展示。

**依赖**：I47。

#### I49：多生态软件源管理

**用户价值**：用户可以统一诊断并恢复 npm、pnpm、Maven、pip、WinGet 和 apt 的源配置。

**内容**：

- 为每种管理器独立增加只读诊断，再逐个开放受控切换。
- 统一可达性、TLS、同步新鲜度、可信度、备份与恢复语义。
- 区分用户级、项目级和系统级配置范围。

**测试/验收**：

- 每个管理器必须先通过只读增量验收才能开放写入能力。
- 项目级源不会被用户级换源计划静默覆盖。
- 凭据和私有仓库地址按隐私策略脱敏。
- 切换失败时原配置可恢复，且不关闭签名或 TLS 校验。

**依赖**：I48。

#### I50：Full Profile 与资源预算

**用户价值**：用户可以组合大多数开发场景，并在执行前了解磁盘、时间、冲突和许可证成本。

**内容**：

- Full 作为模块组合器，不复制各模块清单。
- 增加冲突消解、共享依赖去重、资源预算和分批执行。
- 支持用户从 Full 计划中排除具体模块或工具。

**测试/验收**：

- Full 解析结果等于所选模块的规范化并集。
- 重复工具只保留一个可解释实现。
- 磁盘不足、互斥工具和必须重启步骤在执行前被发现。
- 用户可保存裁剪后的 Profile，并在二次解析时得到相同结果。

**依赖**：I49。

#### I51：第三方适配器 SDK 与兼容测试套件

**用户价值**：社区可以在不修改核心的情况下增加工具支持，同时受到统一契约和安全限制。

**内容**：

- 发布适配器能力声明、统一输入输出、风险规则和测试套件。
- 定义进程隔离、权限、签名/信任和失败降级边界。
- 提供一个只读示例适配器。

**测试/验收**：

- 不合规输出、超时、崩溃和越权请求不会影响核心进程。
- 第三方适配器不能静默降低风险等级。
- 兼容测试套件能在独立仓库运行。
- 用户可以禁用、审查和移除第三方适配器。

**依赖**：I50。

## 6. 每个增量的按需交付物

每个增量必须有需求卡或等价记录，包含用户价值、范围、非范围、依赖、方案、风险、测试和验收标准。其余交付物按实际影响提供：

1. 长期影响架构、安全、兼容性或产品范围的决定更新到 `docs/PROJECT_DECISIONS.md`。
2. 有实现时提供实现及相称测试。
3. 涉及外部输出解析或稳定报告时提供 fixture/golden report。
4. 影响用户入口时更新用户文档和 CLI 帮助。
5. 涉及安全或隐私时记录影响、失败路径和恢复边界。
6. 完成后记录客观测试结果和维护者验收状态。
7. 只有计划发布时才更新版本和发布记录。

不适用事项直接记录为 N/A，不创建空代码、空 fixture 或空文档。

## 8. Backlog 管理规则

### 8.1 优先级

- P0：影响安全、数据完整性或当前增量验收的工作。
- P1：当前里程碑的核心用户价值。
- P2：增强体验但不阻断主流程。
- P3：未来平台、工具或便利功能。

### 8.2 缺陷处理

- 误删、错误提权、敏感信息泄露、计划与实际不一致均为 P0。
- 版本判断错误、活跃路径判断错误原则上不得带入下一增量。
- 外部工具暂时不可用应降级为明确错误或 Unknown，不能伪装成功。

### 8.3 需求变更

新增需求先判断：

1. 是否是当前增量验收的必要条件；
2. 是否改变安全边界或数据契约；
3. 是否能作为新的独立增量；
4. 是否需要更新 PRD、项目决策记录和支持矩阵。

除 P0 外，需求不得在开发中途无边界扩张当前增量。

## 9. 建议的首轮迭代节奏

增量大小应以复杂度和验收风险为准，不强制每个增量持续相同时间。建议：

- I00–I04：每个增量约 2–5 个工作日。
- 单一只读适配器：约 3–7 个工作日。
- 单一执行能力：约 5–10 个工作日，包含 VM 故障测试。
- 跨平台或高风险能力：必要时拆得更小，不用固定 Sprint 长度强行打包。

第一次可执行迭代只启动 **I00 产品契约冻结**。I00 验收完成后才开始 I01，不同时实现扫描、更新和配装。

## 10. 版本与发布建议

- `0.0.x`：工程基线和内部原型。
- `0.1.0`：macOS 本地只读报告，对应 M1。
- `0.2.0`：版本、EOL、项目约束和建议，对应 M2。
- `0.3.0`：Node 安全更新闭环，对应 M3。
- `0.4.0`：macOS Profile 配装，对应 M4。
- `0.5.0`：清理与换源，对应 M5。
- `0.7.0`：Windows/Linux 核心能力，对应 M6。
- `0.8.0`：Skill/MCP 与 Agent 扩展，对应 M7。
- `1.0.0`：安全审计和稳定性门槛通过，对应 M8。
- `1.x`：按 I42–I51 逐个扩展完整工具、受控批量更新、画像、软件源和适配器生态。

版本号只表示已通过相应验收的能力，不按时间自动提升。
